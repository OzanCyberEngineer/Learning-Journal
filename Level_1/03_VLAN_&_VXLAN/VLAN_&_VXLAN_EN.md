# 03 — VLAN & VXLAN

This document covers Virtual Local Area Network (VLAN) and Virtual Extensible LAN (VXLAN) technologies, which are used to logically isolate, scale, and manage traffic in computer networks. This module examines the transition from traditional Layer 2 segmentation approaches to modern Layer 3 overlay architectures utilized in cloud and data center infrastructures. It details architectural components, security risks, and troubleshooting practices. Upon completing this document, the reader will acquire the competencies necessary to design, validate, and troubleshoot modern network segmentation architectures.

## 1. Overview

Network segmentation is the practice of dividing a large, monolithic network infrastructure into smaller, logically or physically isolated sub-networks for performance, security, and manageability reasons. In traditional networks, a single large broadcast domain allows broadcast traffic (such as ARP requests) generated by any device to flood the entire network. This increases bandwidth consumption and introduces lateral movement risks from a security standpoint.

Logical segmentation technologies, namely VLAN and VXLAN, establish logical boundaries within the network to isolate broadcast domains (broadcast-domain isolation) without altering the underlying physical infrastructure. Traditional network architectures utilize VLANs to achieve isolation at Layer 2. Conversely, the emergence of cloud computing, multi-tenancy, and massive-scale data centers has made VXLAN the industry standard for building virtual Layer 2 networks over Layer 3 infrastructures.

## 2. Why it Matters

In modern infrastructure architectures, network segmentation and tunneling protocols are critical across multiple engineering disciplines:

*   **Network Engineering:** Reduces physical cabling and hardware costs while enabling flexible and dynamic network topologies. It eliminates Spanning Tree Protocol (STP) limitations in data centers, allowing all physical links to be utilized simultaneously via Equal-Cost Multi-Pathing (ECMP).
*   **Cybersecurity and Defensive Security:** Reduces the attack surface. Isolating critical server infrastructures, guest networks, and management interfaces from one another prevents a breach in one sub-network from easily compromising the rest of the environment.
*   **Cloud and Platform Engineering:** Provides the flexibility to assign overlapping IP address spaces (overlapping IPs) to different customers (tenants) or isolated pod networks within container orchestration platforms like Kubernetes and public cloud environments.

## 3. Core Concepts and Terminology

### Physical vs Logical Segmentation
Physical segmentation involves deploying separate physical switches, cables, and routers for each group or department. It is rigid, inefficient, and expensive. Logical segmentation overlays software-defined, label-based rules onto the same physical hardware to create isolated networks.

### VLAN Identifiers (VLAN IDs)
A 12-bit field within an 802.1Q tag that identifies which virtual network a frame belongs to. Mathematically, it supports $2^{12} = 4096$ unique values:
*   **0 and 4095:** Reserved for system use; cannot be assigned to user ports.
*   **1:** The default VLAN. All ports are assigned to this VLAN out of the box. It cannot be deleted or renamed.
*   **2 - 1005:** The normal VLAN range, used for day-to-day network operations.
*   **1006 - 4094:** The extended VLAN range. Typically preferred by service providers and large data centers; it cannot be stored in the VLAN Trunking Protocol (VTP) database unless VTP is set to transparent mode.

### Access Ports vs Trunk Ports
*   **Access Port:** A port configured to serve only a single VLAN. End-user devices (PCs, printers, server data interfaces) connect here. Frames exiting an access port adhere to the standard Ethernet structure and do not contain a VLAN tag.
*   **Trunk Port:** A switch-to-switch or switch-to-router link configured to carry traffic for multiple VLANs over a single physical line. It relies on tagging mechanisms to keep traffic separated.

### IEEE 802.1Q and VLAN Tag Structure
IEEE 802.1Q is the industry-standard tagging protocol for trunk links. It inserts a **4-byte (32-bit)** header between the Source MAC address and the Type/Length fields of the original Ethernet frame:

```
+-------------------+---------------------+-----------------+---------+
| Dest/Src MAC (12B)|  802.1Q Tag (4B)    | Type/Length (2B)| Data... |
+-------------------+---------------------+-----------------+---------+
                        |
                        +--> TPID (16 bits): 0x8100
                        +--> TCI  (16 bits):
                              |--> PCP (3 bits)
                              |--> DEI (1 bit)
                              |--> VLAN ID (12 bits)
```

*   **TPID (Tag Protocol Identifier - 16 bits):** Set to `0x8100` to indicate that the frame is encapsulated with an 802.1Q tag.
*   **TCI (Tag Control Information - 16 bits):** Divided into three sub-fields:
    *   **PCP (Priority Code Point - 3 bits):** Specifies the Class of Service (CoS) priority level (0-7) used for Quality of Service (QoS) mechanisms.
    *   **DEI (Drop Eligible Indicator - 1 bit):** Indicates if the frame can be dropped preferentially during network congestion.
    *   **VLAN ID (VID - 12 bits):** The unique identification number of the logical network.

### Tagged and Untagged Frames
*   **Tagged Frame:** A frame that contains the 4-byte 802.1Q header. These traverse trunk lines.
*   **Untagged Frame:** A standard Ethernet frame without a VLAN header. These are processed within access ports and end-user devices.

### Native VLAN
A configuration rule defining which VLAN should receive untagged frames arriving on a trunk port. By default, it is set to VLAN 1 across most vendors. It carries control traffic such as CDP, LLDP, STP, and DTP. Native VLAN IDs must match on both ends of a trunk link.

### Voice VLAN Fundamentals
A hybrid access port mechanism that transmits both IP Phone (VoIP) voice traffic and computer data traffic over a single physical interface. The switch instructs the IP phone (via CDP or LLDP) to tag its voice packets with the designated Voice VLAN ID, while the PC connected to the back of the phone transmits its data untagged. This ensures voice traffic receives QoS prioritization while maintaining data isolation.

### VXLAN Concepts: Underlay, Overlay, VNI, VTEP
*   **Underlay Network:** The physical infrastructure consisting of switches, routers, and cables. Its sole responsibility is to route IP packets reliably and quickly using Layer 3 routing protocols (e.g., OSPF, BGP, IS-IS).
*   **Overlay Network:** A logical, topology-independent virtual tunnel network built on top of the underlay. VXLAN operates in this layer, tunneling Layer 2 frames inside Layer 3 packets.
*   **VNI (VXLAN Network Identifier):** A **24-bit** field that replaces the traditional 12-bit VLAN ID. It supports up to $2^{24} = 16,777,216$ unique virtual networks, resolving the traditional scaling limitation.
*   **VTEP (VXLAN Tunnel Endpoint):** The entity responsible for originating and terminating VXLAN tunnels. It can be implemented as a hardware switch ASIC or as a software module within a hypervisor. A VTEP encapsulates incoming Layer 2 frames with a VXLAN header inside a Layer 3 packet, or decapsulates incoming tunnel packets to forward them to the local Layer 2 network.

## 4. Working Principle

### Inter-VLAN Routing
Devices inside different VLANs cannot communicate directly at Layer 2. A Layer 3 device (a router or a Layer 3 switch) must route traffic between them.

#### Router-on-a-Stick (RoaS)
A legacy design where a single physical interface on a router connects to a trunk port on a switch. The router interface is carved into logical subinterfaces, with each subinterface assigned a default gateway IP address for a specific VLAN. Traffic travels from the switch to the router, is routed, and returns over the same physical link with a new VLAN tag. This architecture introduces a single point of congestion (bottleneck).

#### SVI (Switched Virtual Interfaces) and Layer 3 Switching
Modern enterprise architectures perform Inter-VLAN routing natively within Layer 3 switches via logical interfaces called SVIs (`interface Vlan <ID>`). Routing occurs at wire-speed inside the switch’s hardware ASICs (Application-Specific Integrated Circuits), eliminating the latency and bottleneck constraints associated with the RoaS model.

### VXLAN Forwarding and Learning Mechanisms
VXLAN utilizes a "MAC-in-UDP" encapsulation scheme at the data plane. A VTEP can discover which remote VTEP is hosting a destination MAC address through three primary methods:

#### Flood-and-Learn
A traditional data plane learning mechanism. When a VTEP receives a frame with an unknown destination MAC address (Unicast/Broadcast/Multicast), it encapsulates it and broadcasts it to a predefined IP Multicast group in the underlay network. All VTEP members of that group receive and decapsulate the packet, learning the source MAC-to-VTEP IP mapping and storing it in their local tables. This method does not scale efficiently and generates unnecessary network overhead.

#### Head-end Replication (Ingress Replication)
Used when the underlay network does not support multicast. Instead of using a multicast group, the ingress VTEP replicates the broadcast/multicast frame into multiple unicast packets, sending one individually to each remote VTEP IP defined in its static or dynamically discovered list.

#### EVPN and BGP EVPN Fundamentals (Control Plane Optimization)
Modern data center networks replace "Flood-and-Learn" with a control plane protocol: **MP-BGP EVPN (Multiprotocol BGP Ethernet VPN)**. BGP EVPN enables VTEPs to advertise host MAC and IP addresses to one another via BGP routing updates (**MAC/IP Advertisement - Route Type 2**). This control-plane distribution eliminates the need for network-wide flooding to locate unknown unicast destinations.

## 5. Architecture and Components

### Traditional Campus Network vs Spine-Leaf Architecture
Traditional three-tier architectures (Core, Distribution, Access) are optimized for client-to-server traffic patterns (North-South traffic). They rely heavily on Spanning Tree Protocol (STP) to prevent loops, which intentionally disables half of the redundant physical links.

Modern data centers prioritize server-to-server communication (East-West traffic). To optimize this path, the industry standard has shifted to a two-tier **Spine-Leaf Architecture**, which serves as the physical underlay for VXLAN.

```
       +-----------------+       +-----------------+
       |  Spine Switch 1 |       |  Spine Switch 2 |  <-- Underlay (Layer 3 / IP Fabric)
       +-----------------+       +-----------------+
          /     \                   /     \
         /       \                 /       \
        /         \               /         \
+---------------+ +---------------+ +---------------+
| Leaf 1 (VTEP) | | Leaf 2 (VTEP) | | Leaf 3 (VTEP) |  <-- VXLAN Boundary (Edge)
+---------------+ +---------------+ +---------------+
    |       \         /       |         |       \
[Server A]  [Server B]    [Server C] [Server D]  [Server E] <-- Overlay (Virtual Layer 2)
```

*   **Spine Layer:** The backbone of the network. Spine switches route packets between Leaf switches. They do not maintain VXLAN state or look up endpoint MAC addresses; they route purely based on the outer IP header.
*   **Leaf Layer:** The aggregation layer where servers, storage arrays, and firewalls connect. Each Leaf switch acts as a **VTEP**. Encapsulation and decapsulation happen here.
*   **ECMP (Equal-Cost Multi-Pathing):** Traffic traversing Leaf switches is distributed across all active Spine links simultaneously, avoiding the blockages seen in STP.

## 6. Step-by-Step Technical Workflow

The step-by-step workflow of Server A ($MAC_A$, $IP_A$) sending a frame to Server B ($MAC_B$, $IP_B$) across VNI 10010 is described below:

```
[Server A] -> (L2 Frame) -> [Leaf 1 (VTEP 1)] -> (VXLAN/UDP/IP) -> [Spine] -> [Leaf 2 (VTEP 2)] -> (L2 Frame) -> [Server B]
```

1.  **Frame Generation:** Server A constructs a standard Layer 2 Ethernet frame targeting $MAC_B$ as the destination and sends it out its interface into the access port of **Leaf 1**.
2.  **VNI Mapping and Lookup:** Leaf 1 receives the frame, identifies its incoming VLAN configuration, and maps it to the corresponding **VNI 10010**. Leaf 1 queries its BGP EVPN routing table and determines that $MAC_B$ is located behind **Leaf 2 (VTEP 2)**.
3.  **VXLAN Encapsulation:** Leaf 1 keeps the original Layer 2 frame intact and wraps it with the following headers:
    *   **VXLAN Header:** An 8-byte header containing the VNI value `10010`.
    *   **Outer UDP Header:** The destination port is fixed to IANA standard **4789**. The source port is calculated dynamically using a hash of the inner frame headers to facilitate ECMP load balancing.
    *   **Outer IP Header:** The Source IP is set to *VTEP 1 Loopback IP*, and the Destination IP is set to *VTEP 2 Loopback IP*.
    *   **Outer Ethernet Header:** The MAC addresses correspond to the next-hop Spine switch interface.
4.  **Underlay Transport:** The packet is sent into the physical IP fabric. The Spine switches evaluate only the Outer IP header, routing the packet toward the VTEP 2 IP address using standard Layer 3 routing lookups.
5.  **Decapsulation:** The packet arrives at Leaf 2 (VTEP 2). Recognizing the outer destination IP as its own and identifying UDP port 4789, the switch directs the packet to its VXLAN decapsulation engine. Outer headers (Ethernet, IP, UDP, VXLAN) are stripped away.
6.  **Delivery:** The unencapsulated, original Layer 2 frame is forwarded out the local access interface assigned to VNI 10010, arriving at Server B. Server B processes a clean Ethernet frame, unaware of the underlying encapsulation process.

## 7. Real-World Use Cases

### Case 1: Enterprise Campus Network Segmentation
*   **Problem:** Running users, guests, and system administrators inside a single broadcast domain introduces packet sniffing vectors and facilitates unauthorized lateral movement.
*   **Approach:** Divide the infrastructure into distinct VLAN paths:
    *   `VLAN 10 - User`: For office workstations.
    *   `VLAN 20 - Server`: For internal enterprise servers.
    *   `VLAN 30 - Guest`: Providing internet-only access for visitors.
    *   `VLAN 99 - Management`: Reserved for switch, router, and firewall management interfaces.
*   **Benefit & Complexity:** Gateway terminations are handled via SVIs on a core Layer 3 switch, where strict Access Control Lists (ACLs) block user network endpoints from attempting connection sessions toward the management layer. This introduces operational overhead as ACLs grow.

### Case 2: Multi-Tenant Cloud Datacenter & Kubernetes CNI Implementations
*   **Problem:** Cloud hosting environments require network isolation for thousands of distinct clients (tenants). The 4096 VLAN limit is insufficient, and tenants frequently require identical, overlapping IP schemas (e.g., `10.0.0.0/24`) without conflicting with one another.
*   **Approach:** Transition the physical fabric to a Layer 3 Spine-Leaf architecture running a VXLAN overlay. In Kubernetes clusters, CNI plug-ins like Calico or Cilium are deployed in VXLAN tunneling mode.
*   **Benefit:** Every tenant or pod subnet is isolated into a unique VNI (e.g., Tenant A uses VNI 100001, Tenant B uses VNI 100002). This provides over 16 million available networks. Overlapping IPs remain contained within their respective encapsulations, preventing address collisions across the fabric.

## 8. Security Perspective

### VLAN Hopping Attacks
An exploit vector where an attacker bypasses logical boundaries to transmit or intercept packets on a destination VLAN they are not authorized to access. This is achieved through two methods:

#### 1. Switch Spoofing
*   **Technical Root Cause:** Switch interfaces left in default states that permit dynamic trunk negotiations via Dynamic Trunking Protocol (DTP).
*   **Mechanism:** An attacker crafts rogue DTP negotiation packets from their endpoint. The switch port changes its state to "Trunk". The attacker can then send and receive tagged frames across all VLANs permitted on that trunk.
*   **Prerequisites:** Target switch port must be left in `dynamic desirable`, `dynamic auto`, or a generic default mode.
*   **Defense:** User interfaces must be explicitly configured as static access interfaces (`switchport mode access`) and DTP negotiation disabled entirely via `switchport nonegotiate`.

#### 2. Double Tagging
*   **Technical Root Cause:** The design behavior where switches strip the outer tag off frames arriving from the Native VLAN over a trunk line without further inspections.
*   **Mechanism:** An attacker crafts a frame with two distinct 802.1Q tags. The outer tag matches the current Native VLAN ID of the attacker's port (e.g., VLAN 1). The inner tag is set to the targeted victim's VLAN ID (e.g., VLAN 10). When the first switch receives the frame, it strips off the outer tag because it matches the Native VLAN and forwards the frame over the trunk link. The secondary switch receives the frame, reads the remaining inner tag (VLAN 10), and forwards it to the victim's broadcast domain. This attack is unidirectional; return traffic does not return to the attacker.
*   **Prerequisites:** The attacker must occupy the same VLAN configured as the Native VLAN for the active trunk links.
*   **Defense:** Change the default Native VLAN ID on all trunk lines to an unused, isolated VLAN ID (e.g., VLAN 999) that is not assigned to any user access interfaces.

### Native-VLAN and Trunk Misconfigurations
*   **Risk:** Mismatched Native VLAN assignments between two ends of a switch-to-switch trunk link (Native VLAN Mismatch).
*   **Impact:** Frames transmitted untagged from VLAN 5 on one switch are parsed as belonging to a different VLAN (e.g., VLAN 10) on the receiving switch. This causes data leakage across segments and introduces Spanning Tree topology instability.

### Private VLAN (PVLAN) Concepts
A sub-segmentation technique used to isolate ports operating inside the exact same VLAN at Layer 2. Port behaviors are classified into three types:
*   **Promiscuous Port:** Can communicate with all other ports in the PVLAN domain (including Isolated and Community ports). Typically assigned to routers, firewalls, or shared gateway resources.
*   **Isolated Port:** Maintains connectivity only with Promiscuous ports. It cannot talk to other Isolated ports inside the same VLAN (e.g., deployed in hotel guest networks or DMZ environments).
*   **Community Port:** Communicates with other community ports inside its designated sub-group and with the Promiscuous ports. It cannot cross-communicate with other community groups or isolated allocations.

### Spanning Tree Relationship
Traditional enterprise fabrics deploy a dedicated Spanning Tree instance per individual VLAN (e.g., Per-VLAN Spanning Tree - PVST+). While this enables VLAN-based traffic path engineering, it increases switch CPU load. In contrast, VXLAN underlay fabrics route traffic at Layer 3, eliminating Layer 2 loop topologies and removing the requirement for STP across the core data center fabric.

### VXLAN Security Vulnerabilities and Limitations
VXLAN encapsulation lacks native authentication or cryptographic encryption capabilities. Tunnels transmit packets in cleartext across the underlay infrastructure. If an attacker gains physical or logical access to the underlay link paths, they can sniff the encapsulated payload. To secure this traffic, engineers must deploy **IPsec** or **MACsec** underlying tunnels to encrypt the data before encapsulation.

## 9. Hardening and Best Practices

To ensure secure and predictable operations, the following tightening controls should be applied:

| Control | Rationale | Scope | Limitation / Trade-off |
| :--- | :--- | :--- | :--- |
| **Disable DTP & Force Static Access Mode** | Mitigates Switch Spoofing attacks and rogue trunk creations. | All end-user deployment interfaces. | Increases administrative overhead by requiring explicit interface configurations. |
| **Alter and Isolate Native VLAN ID** | Prevents Double Tagging exploits and unintentional data leaks. | All Switch-to-Switch Trunk links. | Requires strict configuration consistency across the environment to prevent mismatches. |
| **Shutdown Unused Interfaces** | Eliminates unauthorized physical access vectors. | All dormant switch ports. | Requires manual administrative intervention whenever new physical lines are provisioned. |
| **Configure Jumbo Frames (MTU 9000)** | Eliminates packet fragmentation overhead caused by the 50-54 byte VXLAN header. | Entire Underlay Network Path. | Every interconnected device along the path must support high MTU thresholds. |

## 10. Advantages, Disadvantages, and Trade-offs

### Performance and Scalability
*   **VLAN:** Provides low-latency, hardware-driven switching but is capped at 4096 addressable networks. This constraint restricts large-scale public or private multi-tenant cloud scale out.
*   **VXLAN:** Scales up to 16 million VNIs. However, it introduces a 50-byte encapsulation penalty on every packet, reducing effective data payload efficiency and increasing CPU overhead on older hardware that lacks dedicated ASIC offloading modules.

### Operational Complexity
*   **VLAN:** Simple to configure, validate, and troubleshoot. Standard network decoders (e.g., Wireshark) easily interpret the 802.1Q field natively.
*   **VXLAN:** Introduces multi-layered architectures utilizing complex control planes like BGP EVPN. Isulating faults requires separating underlay routing anomalies from overlay mapping issues.

## 11. Common Misconceptions

### Misconception 1
*   **Flawed Thought:** "Applying VLAN segmentation ensures absolute host-to-host security, removing the requirement for internal firewalls."
*   **Rationale:** A VLAN is merely a Layer 2 boundary. As soon as a packet is routed across an SVI or an external router interface, it will pass freely unless explicit Layer 3/4 Access Control Lists (ACLs) or Layer 7 inspection rules are enforced.
*   **Correct Engineering Approach:** Layer 2 segregation reduces the immediate broadcast domain, but cross-VLAN communication paths must be governed by stateful packet filtering devices.

### Misconception 2
*   **Flawed Thought:** "When using a VXLAN overlay, the performance, stability, and path redundancy of the underlay routers become irrelevant."
*   **Rationale:** The overlay tunnel network depends entirely on the health of the underlying physical IP fabric.
*   **System Impact:** Underlay issues such as packet drops, routing convergence loops, or jitter immediately degrade the performance of the VXLAN tunnels running on top.

## 12. Troubleshooting and Validation

### Systematic Troubleshooting Strategy
1.  **Verify Physical Layer:** Check port operational states, transceiver status, and error counters (`CRC errors`, `input errors`).
2.  **Verify Layer 2 Mac Tables:** Check if the switch is successfully registering host MAC address entries on the targeted interface and matching VLAN context.
3.  **Validate Trunk Operational State:** Inspect switch-to-switch interfaces to confirm trunking status, Native VLAN symmetry, and allowed VLAN listings.
4.  **Validate Tunnels (VXLAN specific):** Test connectivity between underlay VTEP endpoints and confirm that firewall rules permit UDP port 4789.

### Essential Validation Commands (Cisco IOS-XE / NX-OS Focus)

#### `show vlan brief`
*   **Purpose:** Lists all configured VLANs and their assigned physical access interfaces.
*   **Expected Healthy Result:** The targeted port must be explicitly visible next to its assigned VLAN ID. If missing, it may be configured as a trunk or shut down.

#### `show interfaces trunk`
*   **Purpose:** Displays active trunk connections, native configurations, and the allowed VLAN matrix.
*   **Anomalous Sign:** Mismatched native VLAN definitions or missing target VLAN IDs within the `VLANs Allowed and Active` output string.

#### `show mac address-table interface <interface-id>`
*   **Purpose:** Identifies the MAC addresses learned on a specific physical or logical interface.
*   **Use Case:** If an endpoint cannot communicate and its MAC address does not register in this table, look for faults at the physical layer or on the host network interface card.

#### `show nve peers` (NX-OS / VXLAN)
*   **Purpose:** Verifies operational tünel connections established with remote VTEP systems.
*   **Expected Healthy Result:** The remote VTEP IP peer block state must explicitly show as `Up`.

### End-to-End MTU Verification Test
To confirm that the underlay fabric handles encapsulated packets without fragmentation, execute a ping test with the Don't Fragment (DF) bit enabled. The following Linux command tests the path with a 1550-byte packet payload:

```bash
ping -M do -s 1550 <Destination_VTEP_IP>
```
*   **Result Analysis:** Receiving a `Frag needed and DF set` notification indicates that at least one transit link along the physical underlay path is capped at standard 1500 MTU limits. This requires adjusting the MTU across the transit path.

## 14. Production Environment Perspective

*   **Failure Modes:** In BGP EVPN networks, a drop in a BGP session on a Leaf switch stops the distribution of new MAC/IP address updates across the fabric. This can cause "blackholing" conditions for newly provisioned or migrated virtual instances, even if the underlying data plane remains physically connected.
*   **High Availability:** To provide redundancy, end servers typically connect to two distinct Leaf switches simultaneously using Multi-Chassis EtherChannel (vPC - Virtual Port Channel) or EVPN Multi-Homing. In this design, both Leaf switches share an `Anycast VTEP` configuration, allowing them to appear as a single logical VTEP to the rest of the fabric.
*   **Monitoring and Observability:** Traditional SNMP polling is insufficient for tracking encapsulated architectures. Production operations require streaming telemetry frameworks to collect NetFlow or sFlow details from tünel boundaries. This telemetry data is typically aggregated into time-series backends like Prometheus and visualized using **Grafana** to monitor overlay tunnel latency, drop rates, and traffic patterns.

## 15. Relationship with Other Technologies

*   **Linux:** Modern kernels (version 3.7.0 and later) provide native support for VXLAN interfaces. This allows Linux servers to establish software-defined overlay tunnels with one another without requiring specialized network hardware.
*   **Kubernetes:** Multi-node container networking environments utilize CNI engines (e.g., Flannel, Calico, Cilium) configured in VXLAN mode to encapsulate inter-pod traffic across distinct Kubernetes nodes.
*   **Infrastructure as Code (IaC):** Managing large-scale Spine-Leaf topologies involves configuring consistent VLAN, VNI, and BGP EVPN definitions across hundreds of devices. Teams typically automate these deployments using **Terraform** and **Ansible** to push standardized configuration templates.

## 16. What a Junior Should Know

*   Explain that a VLAN provides Layer 2 broadcast domain isolation, whereas VXLAN creates a virtual Layer 2 overlay network across a Layer 3 routed underlay network.
*   Differentiate between an access port (which handles untagged end-device traffic) and a trunk port (which carries tagged multiplexed frames between network nodes).
*   Recall the exact scaling limits: traditional VLANs support up to 4096 segments, while VXLAN scales up to approximately 16.7 million VNIs.
*   State that communication between different VLANs requires a Layer 3 device, such as a router interface or a Switched Virtual Interface (SVI) on a Layer 3 switch.
*   Identify trunk misconfigurations (e.g., Native VLAN mismatches) and missing allowed VLANs as the primary troubleshooting targets during connectivity drops.

## 17. Short Summary

*   Network segmentation divides large broadcast domains to improve network performance, scale, and security.
*   Logical segmentation (VLAN/VXLAN) overlays software-defined isolation onto shared physical hardware, avoiding the rigidity of physical separation.
*   The IEEE 802.1Q standard inserts a 4-byte header into standard Ethernet frames to identify virtual networks on trunk links.
*   Traditional VLAN tracking fields are 12 bits long, capping total network creation at 4096 unique IDs.
*   Access ports strip tags and connect to endpoints, whereas trunk ports retain tags to multiplex traffic across switches.
*   Mismatched Native VLAN settings across trunk connections cause data leaks and Spanning Tree loop calculations.
*   Inter-VLAN routing is performed using legacy Router-on-a-Stick subinterfaces or wire-speed Switched Virtual Interfaces (SVIs).
*   VLAN Hopping attacks exploit default DTP behaviors (Switch Spoofing) or native tag parsing rules (Double Tagging).
*   VXLAN utilizes a 24-bit VNI tracking header, expanding available isolation identifiers to over 16 million networks.
*   VXLAN abstracts the physical routing layer (Underlay) from the encapsulated virtual tunnel endpoints (Overlay).
*   VTEP interfaces handle data plane encapsulation and decapsulation over UDP port 4789.
*   MP-BGP EVPN acts as a modern control plane for VXLAN, exchanging endpoint MAC and IP mappings to minimize network flooding.
*   VXLAN adds a 50-54 byte encapsulation overhead, making the implementation of Jumbo Frames (MTU 9000) a best practice in production environments.
